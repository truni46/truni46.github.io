1. Mô tả vòng thi

Đây là vòng thi đầu tiên của Ontological Reasoning in Medical Knowledge Retrieval.

Ở vòng này, thí sinh:

  • Xây dựng pipeline xử lý văn bản y khoa tự do, phát hiện và chuẩn hóa khái niệm y tế.
  • Chạy pipeline trên tập test public do BTC cung cấp.
  • Nộp file kết quả dự đoán theo đúng format quy định.

Top 8 đội có kết quả vòng public cao nhất sẽ được chọn vào Vòng 2.

4. Yêu cầu submission

Thí sinh nộp một file output.zip chứa dự đoán cho toàn bộ tập test.

4.1 Cấu trúc file submission

Sau khi giải nén, cấu trúc:

output/
├── 1.json      # Nhãn dự đoán cho bản ghi 1
├── 2.json      # Nhãn dự đoán cho bản ghi 2
├── ...
└── 100.json

Mỗi file N.json là dự đoán tương ứng với file N.txt trong tập test.

4.2 Format file .json

Mỗi file là một list các dictionary, mỗi dictionary là một khái niệm y tế phát hiện được. Ví dụ thực tế:

Input (1.txt):

"Danh sách thuốc trước nhập viện chính xác và đầy đủ. 1. amlodipine 10 mg po daily 2. aspirin 81 mg po daily 3. metoprolol succinate xl 50 mg po daily 4. guaifenesin ml po q6h:prn điều trị ho 5. nystatin oral suspension 5 ml po qid:prn điều trị đau nhức..."

Output (1.json):

[
  {
    "text": "amlodipine 10 mg po daily",
    "type": "THUỐC",
    "candidates": ["308135"],
    "assertions": ["isHistorical"],
    "position": [58, 83]
  },
  {
    "text": "aspirin 81 mg po daily",
    "type": "THUỐC",
    "candidates": ["243670"],
    "assertions": ["isHistorical"],
    "position": [89, 111]
  },
  {
    "text": "ho",
    "type": "TRIỆU_CHỨNG",
    "assertions": [],
    "position": [196, 198]
  },
  {
    "text": "đau nhức",
    "type": "TRIỆU_CHỨNG",
    "assertions": [],
    "position": [254, 262]
  }
]

4.3 Quy tắc bắt buộc

  • File name khớp: file 1.json ứng với 1.txt, file 2.json ứng với 2.txt,...
  • Trường type: chỉ nhận một trong các nhãn TRIỆU_CHỨNG, TÊN_XÉT_NGHIỆM, KẾT_QUẢ_XÉT_NGHIỆM, CHẨN_ĐOÁN, THUỐC.
  • Trường position: list 2 số nguyên [start, end], index ký tự từ 0 đến n-1.
  • Trường assertions: chỉ nhận giá trị isNegated, isFamily, isHistorical. List rỗng [] nếu không có. Áp dụng cho CHẨN_ĐOÁN, THUỐC, TRIỆU_CHỨNG.
  • Trường candidates:
    • Với CHẨN_ĐOÁN: list mã ICD-10 (string).
    • Với THUỐC: list mã RxNorm (string).
    • Với các loại khác: có thể bỏ trường này hoặc để list rỗng [].

Lưu ý quan trọng: Phân biệt cẩn thận giữa TÊN_XÉT_NGHIỆM (vd: "WBC", "NEUT%") và KẾT_QUẢ_XÉT_NGHIỆM (vd: "14,43", "76,4") — đây là hai khái niệm tách biệt với position khác nhau, không gộp chung.

5. Kiểm tra chống gian lận

Lưu ý quan trọng: Trước khi vòng 1 kết thúc, BTC yêu cầu top ~15 đội gửi trước source code riêng để thực hiện dựng lại và đánh giá trên dữ liệu private test. Việc này nhằm tránh tình trạng gian lận nộp file hard-code output với input được cung cấp.

5.1 Source code phải gồm

  • Tất cả các file code của nhóm (data processing, training, inference,...).
  • Data nhóm sử dụng (kể cả synthetic data, augment data).
  • Model weights.
  • File README hướng dẫn cài đặt chi tiết.

5.2 Quy trình

  • BTC nhận source code và thực hiện cài đặt theo README.
  • Nếu BTC không thể cài đặt được code của nhóm → nhóm sẽ được liên lạc riêng để hỗ trợ trong một khoảng thời gian nhất định.
  • Nếu nhóm không cung cấp hỗ trợ kịp thời → bị loại khỏi vòng tiếp theo.

Hãy chuẩn bị code có thể tái lập (reproducible) ngay từ đầu — fix random seed, ghi rõ dependencies, đóng gói model weights, tránh dependency vào path tuyệt đối hoặc tài khoản cloud cá nhân.

7. Lưu ý riêng cho vòng 1

Về dữ liệu

  • Không có tập train — đây là điểm khác biệt lớn. Thí sinh phải đầu tư nghiêm túc vào xây dựng data huấn luyện:
    • Synthetic data sinh từ LLM lớn.
    • Pretrained medical models (BioBERT, ClinicalBERT, MedBERT,...).
    • Datasets công khai: i2b2, MIMIC, n2c2, BioASQ,...
    • Annotation thủ công một seed nhỏ để fine-tune hoặc làm few-shot.
  • 30 ngày là thời gian dài — tận dụng để thử nghiệm nhiều cách tạo data và pipeline khác nhau.

Về format output

  • Đọc kỹ format output trong đề bài chính trước khi viết pipeline:
    • position là vị trí ký tự (character index), không phải vị trí token.
    • assertions chỉ nhận 3 giá trị hợp lệ (isNegated, isFamily, isHistorical).
    • candidates chỉ áp dụng cho CHẨN_ĐOÁN và THUỐC.
  • Phân biệt TÊN_XÉT_NGHIỆM vs KẾT_QUẢ_XÉT_NGHIỆM: phải tách riêng tên và giá trị (vd: "WBC" và "14,43" là 2 khái niệm), không gộp lại.

Về các trường hợp đặc biệt

  • Văn bản y khoa Việt Nam có nhiều viết tắt (WBC, NEUT%, tbm,...), lỗi chính tả, thuật ngữ địa phương, ký hiệu chuyên ngành (vd: "po", "qid:prn", "q6h").
  • Mapping ICD/RxNorm thường có nhiều mã candidate cho cùng một khái niệm — pipeline nên trả về danh sách candidate theo thứ tự độ tin cậy giảm dần.
  • Phân loại type chính xác rất quan trọng — đoán đúng text nhưng sai type (vd: nhầm TRIỆU_CHỨNG thành CHẨN_ĐOÁN) ảnh hưởng lớn đến điểm.

Về vận hành

  • Validate file JSON trước khi nộp: file lỗi cú pháp JSON, sai key, sai kiểu dữ liệu đều có thể làm điểm bằng 0 cho file đó.
  • Chuẩn bị sẵn source code reproducible từ đầu — không phải đến cuối vòng mới đóng gói.
  • 5 submit/ngày — chọn submit thông minh, không spam thử nghiệm vụn vặt.
  • Mọi thắc mắc về dữ liệu hoặc submission liên hệ kênh hỗ trợ chính thức của BTC.
